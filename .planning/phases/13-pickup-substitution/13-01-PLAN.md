  <action>
    1. In `FfEncoder.ResolveEncoding` (line ~370), add a case for 24-bit:
       ```
       24 => (AVCodecID.AV_CODEC_ID_PCM_S24LE, AVSampleFormat.AV_SAMPLE_FMT_S32)
       ```
       Note: FFmpeg has no native 24-bit sample format; PCM_S24LE codec with S32 input samples truncates to 24-bit in the WAV container. This is the standard FFmpeg pattern for 24-bit encoding.

    2. Extend `AudioInfo` record struct in `AudioProcessor.cs` (line ~208) to add a `BitsPerSample` field:
       ```csharp
       public readonly record struct AudioInfo(
           string Format, int SampleRate, int Channels, TimeSpan Duration, int BitsPerSample);
       ```

    3. In `FfDecoder.Probe` (line ~65), extract bit depth from the codec context. FFmpeg exposes `codecpar->bits_per_raw_sample` for PCM codecs (24 for pcm_s24le, 16 for pcm_s16le) and `codecpar->bits_per_coded_sample` as fallback. If both are 0, infer from `codecpar->format` (AV_SAMPLE_FMT_S16=16, AV_SAMPLE_FMT_S32/FLT=32, AV_SAMPLE_FMT_FLTP=32). Pass the resolved bit depth to the AudioInfo constructor.

    4. In `PolishService.PersistCorrectedBuffer`, before calling `AudioProcessor.EncodeWav`, probe the source chapter WAV to get its bit depth:
       ```csharp
       // Determine source bit depth for format-preserving encode
       var treatedPath = Path.Combine(descriptor.RootPath, $"{descriptor.ChapterId}.treated.wav");
       var sourcePath = File.Exists(correctedPath) ? correctedPath : treatedPath;
       int? sourceBitDepth = null;
       if (File.Exists(sourcePath))
       {
           var info = AudioProcessor.Probe(sourcePath);
           sourceBitDepth = info.BitsPerSample > 0 ? info.BitsPerSample : null;
       }
       var options = new AudioEncodeOptions(
           TargetSampleRate: buffer.SampleRate,
           TargetBitDepth: sourceBitDepth);
       AudioProcessor.EncodeWav(correctedPath, buffer, options);
       ```
       Replace the existing `AudioProcessor.EncodeWav(correctedPath, buffer)` call (which defaults to 16-bit) with this format-preserving version.

    5. The StreamingEncoderSink in FfEncoder also uses ResolveEncoding (line ~554) -- verify it picks up the new 24-bit case automatically (it should, since both codepaths call the same static method).

    Add three static helper methods to AudioSpliceService for roomtone operations:

    1. **GenerateRoomtoneFill** -- Loop a roomtone buffer to fill a target duration:
       ```csharp
       public static AudioBuffer GenerateRoomtoneFill(AudioBuffer roomtone, double targetDurationSec)
       ```
       - Calculate target sample count from targetDurationSec * roomtone.SampleRate
       - If roomtone is already long enough, trim to exact target length via AudioProcessor.Trim
       - If shorter, loop: copy roomtone.Planar[ch] into result repeatedly until filled, then trim excess
       - Preserve roomtone.Metadata on the result
       - Handle edge case: targetDurationSec <= 0 returns empty buffer

    2. **DeleteRegion** -- Remove a time range and crossfade the edges together:
       ```csharp
       public static AudioBuffer DeleteRegion(
           AudioBuffer original, double startSec, double endSec, double crossfadeSec = 0.030, string curve = "tri")
       ```
       - Trim before segment: AudioProcessor.Trim(original, TimeSpan.Zero, TimeSpan.FromSeconds(startSec))
       - Trim after segment: AudioProcessor.Trim(original, TimeSpan.FromSeconds(endSec))
       - If before or after is empty/negligible, return the other
       - Otherwise crossfade: use FfFilterGraphRunner with acrossfade filter to join before+after (same pattern as ReplaceSegment but without the replacement in between)
       - This avoids the zero-length buffer issue identified in Research Pitfall 5

    3. **InsertAtPoint** -- Insert audio at a single time point (zero-width splice):
       ```csharp
       public static AudioBuffer InsertAtPoint(
           AudioBuffer original, double insertPointSec, AudioBuffer insertion, double crossfadeSec = 0.030, string curve = "tri")
       ```
       - Trim before: AudioProcessor.Trim(original, TimeSpan.Zero, TimeSpan.FromSeconds(insertPointSec))
       - Trim after: AudioProcessor.Trim(original, TimeSpan.FromSeconds(insertPointSec))
       - Crossfade before+insertion, then crossfade result+after
       - Or use two sequential ReplaceSegment calls
       - This avoids the start==end validation issue identified in Research Pitfall/Open Question 2

    For all methods: follow the existing pattern (static, use AudioProcessor.Trim, FfFilterGraphRunner for crossfade). Add XML doc comments.
