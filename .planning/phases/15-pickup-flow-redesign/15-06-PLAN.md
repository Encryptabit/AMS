**1. Projection-mapped region positions:**

When displaying waveform regions for staged/committed replacements, positions must use TimelineProjection:

```csharp
// OLD: used raw OriginalStartSec/OriginalEndSec (which were mutated by ShiftDownstream)
// NEW: map baseline positions through projection
var currentStart = StagingQueueService.GetCurrentTime(chapterStem, replacement.OriginalStartSec);
var currentEnd = StagingQueueService.GetCurrentTime(chapterStem, replacement.OriginalEndSec);
```

Update `RefreshRegions` (or equivalent region-sync method) to map ALL region positions through the projection before calling `AddEditableRegion` / `AddRegion` on the waveform.

**2. Chapter-side boundary editing (existing — verify works with projection):**

The existing green draggable regions on the chapter waveform should continue to work. When the user drags a region handle:
- The callback receives the NEW position in CURRENT chapter time
- Convert back to baseline time for storage: this requires an inverse projection (or store the delta)
- **Simpler approach per locked decision "user has final say":** Store the user-adjusted boundaries directly on the StagedReplacement via `UpdateBoundaries`. Since these are baseline coordinates and the user is adjusting in current-time space, use a reverse mapping: `baselinePos ≈ currentPos - projectionOffset`. The exact inverse isn't needed because the user is setting the final position — just store the user's chosen current-time position and note it overrides the baseline.

**Actually, the cleanest approach:** The StagedReplacement stores baseline coordinates. The region displays at projected positions. When the user drags, the callback receives current-time positions. To update the staging queue: `newBaseline = oldBaseline + (newCurrent - oldCurrent)`. This maintains the baseline-coordinate invariant while allowing user adjustment.
