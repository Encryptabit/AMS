<task type="auto">
  <name>Task 2: Simplify UndoService for rebuild-based revert</name>
  <files>host/Ams.Workstation.Server/Services/UndoService.cs</files>
  <action>
Read the current UndoService.cs (322 lines) and adapt for rebuild-based revert:

**Current behavior:** UndoService saves original audio segments as versioned WAV files (`sent{id}.v{N}.original.wav`) before each replacement is applied. On revert, it restores the original segment via surgical splice-back.

**New behavior for rebuild-based revert:**
1. **Still save original segments** — we need them for the case where ALL edits are reverted (we need the pristine treated audio)
2. **Also save replacement audio segments** — for rebuild, we need to re-apply edits to the original treated audio, which means we need the pickup audio that was used for each replacement

Add a new method:
```csharp
public async Task SaveReplacementSegmentAsync(
    string chapterStem,
    string replacementId,
    AudioBuffer replacementBuffer,
    CancellationToken ct)
```
This saves the pickup's trimmed audio to `{undoDir}/sent{sentenceId}.{replacementId}.replacement.wav` so the rebuild process can load it.

Add a retrieval method:
```csharp
public AudioBuffer? LoadReplacementSegment(string chapterStem, string replacementId)
```

**Keep:** The existing `SaveOriginalSegmentAsync` and `RestoreOriginalSegmentAsync` — they're still needed for:
- Saving the per-sentence original segment before first edit
- Providing original segment data if the rebuild approach needs individual segments
- The manifest tracking system

**Remove/simplify:**
- The surgical splice-back logic in `RestoreOriginalSegmentAsync` is no longer the primary revert path. Keep the method but rename conceptually — it now just loads the backed-up segment data rather than performing the splice. The actual rebuild happens in PolishService (Plan 05).

**Update manifest format** to include replacement segment paths alongside original segment paths.
